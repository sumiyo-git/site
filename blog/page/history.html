blog/ 路径
当未找到资源时，Cloudflare 会试图发送 index.html 或 404.html（并非简单的重定向）。本项目中，不存在 root/blog.html、root/blog/index.html 和 root/404.html文件，对于所有 https://sumiyo.link/blog 请求，服务器实则发送 root/index.html，这就是 blog/ 路径的工作原理。

相关文档: https://developers.cloudflare.com/pages/configuration/serving-pages/

2024-08-29

添加留言时，改用请求体传递留言的昵称和内容
试了不下 50 次，才发现读取请求体的写法是 context.request.json() ... 原先要把数据 Base64 编码后（header 仅支持英文、数字和部分字符），挂到请求头上发送，再用 Cloudflare 的官方文档里的 context.request.headers.get() 方法读取参数。
Cloudflare Functions
可以把他简单理解为随 Pages 一起部署的 Worker（实际上代码写法有所不同），这个主要用来实现静态 Pages 页面和后端的互交，比如读写 D1 数据库、KV 等等。这里主要讲讲传参的问题。
// 假如有一个带有 header 和 body 的请求，用以下 Functions 获取请求中的参数 // 文件: root/functions/debug.js export async function onRequest(context) { // 严格模式下，禁止隐式创建全局变量，也就是说直接写 token = 1 会报错 // 获取 header 中的 'Token' 值 var token = context.request.headers.get('Token') // 获取整个 body（结果直接是 json 对象） var body = context.request.json() // 获取 body 中以 ArrayBuffer 形式传递的单个文件 var file = context.request.arrayBuffer() }
